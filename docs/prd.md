# 客户雷达 — 产品需求文档 (PRD)

> 版本: v1.1 | 日期: 2026-07-03 | 状态: 含爬虫策略模式

---

## 1. 项目背景

### 1.1 行业痛点

中国大陆的招投标信息散落在**成千上万个网站**上，企业获取商机面临三大核心难题：

1. **信息分散**：政府平台、代理机构、企业官网、聚合网站各自为政，人工逐一监控不现实。
2. **信息盲区**：聚合平台（千里马、剑鱼等）能覆盖公开招标，但对企业自主采购、邀请招标、小额询价等"内部信息"几乎无法触达。
3. **数据噪声大**：原始公告包含大量 HTML 标签、导航栏、页脚免责声明，人工阅读效率低，传统关键词匹配准确率差。

### 1.2 项目目标

构建一套 **AI 原生的招投标信息采集与商机匹配系统**，实现：

- **广度覆盖**：通过聚合平台全网监控公开招标信息
- **深度穿透**：对接企业自建采购平台，突破聚合平台的信息盲区
- **智能提纯**：利用 LLM 自动清洗、摘要、打标，将原始公告转化为结构化商机数据
- **语义检索**：支持 RAG 语义搜索，让用户用自然语言找到最匹配的项目

## 2. 招投标信息源分类

### 2.1 四大类平台

| 类型 | 说明 | 代表平台 | 数据获取方式 |
|---|---|---|---|
| **第一类：政府公共资源交易/政采平台** | 最权威的法定公开渠道。所有政府机关、事业单位、使用财政资金的项目必须在此公开招标。 | 中国政府采购网、各省公共资源交易中心 | 免费公开，爬虫可抓取 |
| **第二类：招标代理机构网站** | 第三方招标代理公司受委托发布的项目。代理客户涵盖国企、政府、私企。 | 国信招标、中化商务、中国通用招标 | 通常公开，爬虫可抓取 |
| **第三类：企业自主招标/采购平台** | 大型企业自建的 SRM 或电子招投标平台。需先注册入库才能查看。 | 国家电网电子商务平台、中石油电子招标平台、中煤招标网 | 封闭系统，需登录/入库 |
| **第四类：第三方聚合平台** | 商业网站利用爬虫聚合全网招标信息，提供搜索和推送服务。通常为收费会员制。 | 千里马招标网、剑鱼招标、标讯网 | 会员制，API 或页面抓取 |

### 2.2 信息盲区分析

聚合平台**无法覆盖**的场景：

| 盲区类型 | 原因 | 影响范围 |
|---|---|---|
| **企业封闭 SRM** | 不登录不给看；反爬技术严密；企业 IT 团队主动封锁 | 大型央企、国企、知名民企的内部采购 |
| **民企自主采购** | 腾讯、阿里、华为等纯民营企业不受《招标投标法》约束，采购信息不公开 | 知名民企的供应链项目 |
| **小额采购** | 国企内部小额项目走邀请招标/三方询价，只在内部供应商群发布 | 几十万级别的设备采购、软件服务 |

### 2.3 双轨数据采集策略

| 轨道 | 目标 | 数据源 | 典型场景 |
|---|---|---|---|
| **广度轨道** | 全网公开招标不漏单 | 第一类（政采平台）+ 第四类（聚合平台） | 政府采购、学校医院、国企大型公开招标 |
| **深度轨道** | 突破信息盲区 | 第二类（代理机构）+ 第三类（企业官网） | 央企内部采购、定向邀请招标、供应商专属项目 |

> **核心原则**：找线索靠广度轨道（聚合平台设关键词全网跑），锁定目标后靠深度轨道（去源头官网下载标书、投标）。

## 3. 目标用户与使用场景

### 3.1 用户角色

| 角色 | 描述 | 核心需求 |
|---|---|---|
| **商务人员** | 负责市场开拓、投标的前线人员 | 快速发现匹配商机，不漏单 |
| **销售管理者** | 负责团队任务分配、市场策略 | 掌握市场动态、竞对分析 |
| **AI Pipeline 运维** | 负责系统维护、数据质量 | 监控采集状态、处理异常 |

### 3.2 核心使用场景

**场景 A：日常商机监控**
> 商务人员登录系统，查看今日新增的匹配公告，按行业/地区/金额筛选，快速判断是否跟进。

**场景 B：语义搜索**
> 用户输入"广东省内需要涉密资质的华为鸿蒙开发项目"，系统通过 RAG 语义检索返回最相关的公告。

**场景 C：竞对分析**
> 查看某个竞争对手在哪些项目中出现过（通过 AI 标签中的"潜在竞争对手"字段）。

**场景 D：数据质量监控**
> AI Pipeline 运维人员查看 `ai_status` 分布，处理失败任务，调整低效平台的 extraction_prompt。

## 4. 功能需求

### 4.1 数据采集层

| 功能 | 优先级 | 描述 |
|---|---|---|
| 平台画像管理 | P0 | 维护每个数据源平台的技术特征 + 爬虫策略（见 4.5 策略模式） |
| 爬虫调度引擎 | P0 | 读取 `spider_strategy` 字段，分发给对应的爬虫核心 |
| 增量采集 | P0 | 基于 `source_unique_id` 去重，只入库新公告 |
| 企业官网对接 | P1 | 对接 20-30 家目标企业的供应商门户，需人工注册入库后对接 |
| 代理机构监控 | P1 | 针对重点代理机构网站设置采集任务 |
| 聚合平台对接 | P2 | 对接千里马/剑鱼等平台的搜索 API 或页面 |

### 4.2 AI 数据处理层

| 功能 | 优先级 | 描述 |
|---|---|---|
| HTML 清洗 | P0 | BeautifulSoup 提取纯文本，写入 `cleaned_content`，`ai_status → 1` |
| 摘要生成 | P0 | LLM 提取 200 字商机摘要，写入 `notice_summary`，`ai_status → 2` |
| 结构化打标 | P0 | LLM JSON Mode 提取标签（技术关键词、资质要求、行业、竞争对手），写入 `notice_tag`，`ai_status → 3` |
| 向量化 | P1 | 对摘要调用 embedding API，写入 `notice_embedding`，`ai_status → 4` |
| 垃圾识别 | P1 | AI 判断无效公告（广告、过期、重复），标记 `ai_status → -1` |
| 失败重试 | P1 | 处理失败的记录标记 `ai_status → -2`，支持人工干预后重跑 |
| 平台级 Prompt 定制 | P2 | 不同平台使用不同提取 Prompt，存储在 `platform_source.extraction_prompt` |

### 4.3 查询与展示层

| 功能 | 优先级 | 描述 |
|---|---|---|
| 公告列表浏览 | P0 | 按日期、地区、类型、金额筛选，支持分页 |
| 关键词搜索 | P0 | 基于 pg_trgm 的标题和摘要模糊搜索 |
| 标签筛选 | P1 | 按 AI 标签类型和值筛选（如资质=CS4级） |
| 语义搜索 | P2 | 输入自然语言描述，通过向量相似度返回匹配公告 |
| 商机推荐 | P2 | 根据用户画像和历史行为，主动推送匹配公告 |
| 竞对追踪 | P2 | 按竞争对手标签聚合，查看某竞对参与的项目分布 |

### 4.4 系统管理

| 功能 | 优先级 | 描述 |
|---|---|---|
| AI Pipeline 状态监控 | P0 | 仪表盘展示各状态的公告数量分布 |
| 平台管理 CRUD | P0 | 增删改查数据源平台信息，含技术画像和爬虫策略 |
| Prompt 版本管理 | P1 | 查看/回滚平台提取 Prompt 的历史版本 |
| 数据质量报告 | P2 | 统计各平台的数据采集成功率、AI 处理成功率 |

### 4.5 爬虫策略模式 (Spider Strategy Pattern)

核心思路：**认准供应商，写一个模板就能通刷几十个网站**。

中国大陆 70%+ 的电子招投标系统由几家固定软件商开发（新点、博思、广联达等），同一供应商开发的网站接口结构几乎一模一样。系统通过 `platform_source.spider_strategy` 字段实现策略分发：

| spider_strategy | 技术栈 | 对应平台特征 |
|---|---|---|
| `requests_plain` | requests + BeautifulSoup | 服务端渲染，无反爬 |
| `requests_post_xml` | requests + POST 解析 | 新点软件系：POST + viewstate 参数 |
| `requests_with_ja3` | curl_cffi / cloudscraper | 有云盾/加速乐，需模拟 TLS 指纹 |
| `playwright_headless` | Playwright 无头浏览器 | SPA 架构，前端渲染 |
| `playwright_undetected` | Playwright + 反检测 | 企业 SRM，高反爬 + 登录墙 |
| `api_json` | 直接调 JSON API | 有公开或半公开 API |
| `api_with_token` | API + Bearer Token | 企业 SRM，需入库后 Token 访问 |

新增平台时只需填写 `spider_strategy` 和 `spider_config`（JSON），无需改代码。同供应商的多个地市平台共享同一爬虫模板。

#### 4.5.1 广东省主流平台技术画像

种子数据已写入 [002_platform_tech_profile.sql](../supabase/migrations/002_platform_tech_profile.sql)，覆盖：

- **政采平台**：广东省政采网、广州/深圳公共资源交易中心
- **代理机构**：国e平台（省机电招标中心）、国信招标、中化商务
- **企业 SRM**：国家电网、中石油、中煤
- **聚合平台**：千里马、剑鱼
- **新点软件系模板**：佛山/珠海/惠州公共资源交易中心（共享同一爬虫模板）

#### 4.5.2 爬虫配置示例

```json
{
  "platform_name": "广东省机电设备招标中心（国e平台）",
  "spider_strategy": "requests_with_ja3",
  "spider_config": {
    "method": "POST",
    "tls_fingerprint": "chrome120",
    "note": "阿里云盾防护，需模拟JA3指纹"
  }
}
```

```json
{
  "platform_name": "国家电网电子商务平台",
  "spider_strategy": "playwright_undetected",
  "spider_config": {
    "note": "SPA架构+登录墙+自研WAF，需入库后用Token访问",
    "token_location": "localStorage",
    "token_key": "access_token",
    "token_refresh": "manual"
  }
}
```

## 5. 非功能需求

| 维度 | 要求 |
|---|---|
| **数据规模** | 初期支撑 10 万条公告，可扩展至 100 万+ |
| **AI 处理吞吐** | 单次 LLM 调用 < 5 秒，支持批量并发处理 |
| **检索响应** | 关键词搜索 < 500ms，语义搜索 < 2s |
| **数据安全** | Supabase RLS 隔离，service_role 仅后端使用 |
| **Token 成本** | 通过分层存储（只读摘要不读全文）控制 LLM 调用成本 |
| **可追溯** | Prompt 变更、AI 处理状态、错误信息全部可追溯 |

## 7. 数据库设计

详见 [design.md](./design.md) 和 [supabase/migrations/001_init_schema.sql](../supabase/migrations/001_init_schema.sql)。

核心设计原则：

- **分层存储**：`notice_content`（原始）→ `cleaned_content`（清洗）→ `notice_summary`（摘要）→ `notice_tag`（标签），逐级降本
- **AI 状态机**：7 态设计（0→1→2→3→4, -1, -2），支持断点续跑
- **爬虫策略模式**：`spider_strategy` + `spider_config`，新增平台无需改代码
- **标签行级存储**：`notice_tag` 表走 B-Tree 索引，避免 JSON 路径查询的性能瓶颈
- **向量预埋**：`notice_embedding` 表为 RAG 语义检索做准备
- **Prompt 可追溯**：`platform_prompt_history` 记录每次迭代

## 8. MVP 范围与里程碑

### Phase 1: 数据基座 (Week 1-2)
- 在 Supabase 上执行建表迁移（001 + 002）
- 搭建 Python 爬虫调度框架（策略模式：根据 `spider_strategy` 分发）
- 实现 `requests_plain` 策略（最轻量，先跑通政采平台）
- 实现 HTML 清洗 + 去重入库

### Phase 2: AI Pipeline (Week 3-4)
- 实现摘要生成（LLM → `notice_summary`）
- 实现结构化打标（LLM JSON Mode → `notice_tag`）
- 实现 AI 状态机 + 错误处理
- 对接平台级 Prompt 定制

### Phase 3: 前端展示 (Week 5-6)
- 公告列表 + 筛选 + 搜索
- 标签筛选
- AI Pipeline 状态监控仪表盘

### Phase 4: 语义增强 (Week 7-8)
- 向量化 Pipeline
- 语义搜索接口
- 商机推荐

## 9. 开放问题

| # | 问题 | 影响 |
|---|---|---|
| 1 | 首批要对接的目标企业名单（深度轨道）？ | 决定 Phase 1 爬虫开发的优先级 |
