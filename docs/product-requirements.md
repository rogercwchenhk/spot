# 客户雷达 — 产品需求文档 (PRD)

> 版本: v2.0 | 日期: 2026-07-03 | 状态: 基于需求烤问修正

## 1. 项目背景

### 1.1 公司概况

广东励康信息技术有限公司，主营 IT 运维服务：

- **现金牛业务**（最高优先级）：小型机、存储、网络、数据库的维保和运维（IBM Power、Oracle 等）；驻场桌面运维（接转现有人员或新招聘）；交警驻场运维
- **探索业务**：本地算力租赁（DGX SPARK + LLM，如 DeepSeek/Qwen/Gemma + AI 网关 + 前端 AI Agent 部署）；AI 驱动的软件开发

### 1.2 行业痛点

1. **信息分散**：招标公告散落在政府平台、代理机构、企业官网、聚合网站，人工逐一监控不现实
2. **匹配低效**：看到公告后要人工判断"能不能投"——核对资质、人员证书、同类经验、估算扣分，耗时且容易误判
3. **机会遗漏**：没有系统化监控，依赖销售个人经验和人脉，每月能发现的投标机会有限

### 1.3 项目目标

构建一套 **招投标情报采集与商机匹配系统**，以运维/驻场类项目为核心：

- **情报采集**：通过知了标讯 API 获取广东省公开招标公告（政府+企业），逐步扩展到企业自建采购平台
- **智能匹配**：AI 提取公告结构化字段，对照公司资质和人员资质，自动判断"能不能投、扣几分"
- **即时触达**：匹配结果通过企微群机器人 webhook 推送，销售不用主动打开系统
- **竞争分析**：收集中标公告，建立竞争对手档案，分析谁在做哪些同类业务
- **初步目标**：每月至少发现 1 个可投标的机会

## 2. 数据源策略

### 2.1 主源：知了标讯 API

- 已购买 API 服务，有 API Key
- 覆盖广东省政府公共资源交易平台和部分企业招标公告
- API 接口文档：https://ai.zhiliaobiaoxun.com/docs
- 获取方式：按关键词定时拉取，增量入库

### 2.2 辅源：企业自建采购平台（第二阶段）

- 目标：逐步对接 3 个左右广东企业的自建采购平台
- 这些平台不在聚合平台覆盖范围内，需要单独采集
- 企业平台技术栈各异，需逐个分析和开发采集器
- 详见 [platform-registry.md](./platform-registry.md)

### 2.3 不采用的数据源

| 数据源 | 放弃原因 |
|---|---|
| 剑鱼标讯 | 无 API，仅邮件推送，过滤自由度低 |
| 千里马等其他聚合平台 | 知了 API 已覆盖主要需求，ROI 不高 |

### 2.4 数据覆盖验证

> **风险提示**：知了标讯对企业自建平台公告的覆盖范围未知。上线后需用已知的历史项目验证其命中率。如果覆盖率不足，需及时扩展自建平台采集。

## 3. 目标用户与使用场景

### 3.1 用户角色

| 角色 | 描述 | 核心需求 |
|---|---|---|
| **商务人员** | 市场开拓、投标的前线人员（1-2人） | 快速发现匹配商机，知道"能不能投" |
| **销售管理者** | 团队任务分配、市场策略 | 掌握市场动态、竞对分析、趋势 |

### 3.2 核心使用场景

**场景 A：企微群收到新标讯推送**
> 企微群机器人推送一条消息："🟢 强推 | XX单位信息系统运维服务采购 | 预算150万 | 广州 | 截止7月25日 | 预估扣分0分"。商务人员点击链接查看详细匹配结果。

**场景 B：手机 PWA 查看标讯列表**
> 打开手机主屏幕上的"客户雷达"图标，查看今日新增标讯，按匹配等级（强推/可以投/风险/不建议）筛选。

**场景 C：查看匹配详情和扣分明细**
> 打开某条标讯详情，看到 AI 提取的结构化信息（预算、资质要求、评分规则），以及逐项匹配结果（哪些满足、哪些不满足、每项扣多少分）。

**场景 D：竞对分析**
> 查看某竞争对手在哪些项目中出现过，分析其业务覆盖范围。

**场景 E：中标公告回顾**
> 项目投标后记录结果（中标/未中标、原因、得分），用于后续分析和改进。

## 4. 功能需求

### 4.1 数据采集层

| 功能 | 优先级 | 描述 |
|---|---|---|
| 知了 API 对接 | P0 | 通过知了标讯 API 按关键词拉取广东省 IT 运维类公告，增量入库 |
| 增量去重 | P0 | 基于 `source_unique_id` 去重，只入库新公告 |
| 平台画像管理 | P1 | 维护数据源平台的技术特征（为后续爬虫扩展做准备） |
| 企业平台爬虫 | P2 | 逐步对接 3 个左右企业自建采购平台 |

### 4.2 AI 数据处理层

| 功能 | 优先级 | 描述 |
|---|---|---|
| HTML 清洗 | P0 | 提取纯文本，写入 `cleaned_content`，`ai_status → 1` |
| 摘要生成 | P0 | mimo-v2.5-pro 提取 200 字商机摘要，写入 `notice_summary`，`ai_status → 2` |
| 结构化打标 | P0 | mimo-v2.5-pro JSON Mode 提取结构化字段（项目类型、预算、资质要求、评分规则、技术关键词），写入 `notice_tag`，`ai_status → 3` |
| 垃圾识别 | P1 | AI 判断无效公告（广告、过期、非相关），标记 `ai_status → -1` |
| 失败重试 | P1 | 处理失败的记录标记 `ai_status → -2`，支持重跑 |

### 4.3 资质匹配引擎（核心差异化功能）

| 功能 | 优先级 | 描述 |
|---|---|---|
| 公司资质库 | P0 | 存储公司所有资质证书（营业执照、ISO9001、ISO27001、ITSS、CS 等），含等级、有效期 |
| 人员资质库 | P0 | 存储人员证书（PMP、OCP、RHCE、CCIE、HCIE、软考等），含有效期 |
| 资格匹配 | P0 | 将公告的资质要求与公司/人员资质库逐项比对 |
| 扣分预估 | P0 | 客观分逐项计算扣分，主观分按 90% 打折，汇总总扣分 |
| 推荐等级 | P0 | 总扣分 ≤0：强推 / ≤2：可以投 / ≤5：风险 / >5：不建议 |
| 同类项目经验匹配 | P1 | 以运维主要产品线为准（IBM小型机、Oracle数据库等），做过同行业更好 |

**匹配逻辑详情：**

```
同类项目经验判定规则：
- 核心：运维的主要产品类似即可（如都有 IBM 小型机、Oracle 数据库）
- 加分：同行业经验更好（如银行运维 → 做过其他银行的同类产品）
- 公告中不明确写产品型号时（如只写"服务器维保"），AI 需判断并标注不确定性

评分计算规则：
- 客观分：逐项对照评分标准，满足得满分，不满足扣对应分值
- 主观分：按满分的 90% 估算（保守策略，实际可能更低）
- 总扣分 = 客观分扣分之和 + 主观分扣分之和
- 扣分 > 5 分：不建议参加
```

## 5. 爬虫策略模式（第二阶段，为扩展预留）

> 本节为后续企业自建平台扩展做设计预留。MVP 阶段不涉及爬虫开发。

系统通过 `platform_source.spider_strategy` 字段实现策略分发，新增平台只需配置 `spider_strategy` + `spider_config`：

| spider_strategy | 技术栈 | 对应平台特征 |
|---|---|---|
| `requests_plain` | requests + BeautifulSoup | 服务端渲染，无反爬 |
| `requests_post_xml` | requests + POST 解析 | 新点软件系：POST + viewstate |
| `requests_with_ja3` | curl_cffi / cloudscraper | 有云盾/加速乐，需模拟 TLS 指纹 |
| `playwright_headless` | Playwright 无头浏览器 | SPA 架构，前端渲染 |
| `playwright_undetected` | Playwright + 反检测 | 企业 SRM，高反爬 + 登录墙 |
| `api_json` | 直接调 JSON API | 有公开或半公开 API |
| `api_with_token` | API + Bearer Token | 企业 SRM，需入库后 Token |

## 6. 非功能需求

| 维度 | 要求 |
|---|---|
| **数据规模** | 初期支撑 1 万条公告，可扩展至 10 万+ |
| **AI 模型** | 小米 mimo-v2.5-pro |
| **检索响应** | 关键词搜索 < 500ms |
| **数据安全** | Supabase RLS 隔离 |
| **Token 成本** | 通过分层存储控制 LLM 调用成本（只读摘要不读全文） |
| **可追溯** | AI 处理状态、错误信息全部可追溯 |
| **部署** | PWA 移动端优先，可安装到手机主屏幕 |

### 技术选型

| 层 | 选型 | 理由 |
|---|---|---|
| 前端 | React 18 + Vite + shadcn/ui + Tailwind CSS | 国际化视觉质感，源码级样式可控，响应式灵活 |
| 后端 | Node.js + Express | 与已有项目经验一致，一套栈维护成本最低 |
| 数据库 | Supabase (PostgreSQL 15+) | 已有部署经验，RLS/Auth 内置 |
| AI 模型 | 小米 mimo-v2.5-pro | 结构化提取能力强，中文优化好 |
| 定时调度 | node-cron | 轻量，无需额外基础设施 |
| 推送 | 企微群机器人 webhook | 一行代码，销售零门槛 |
| 部署 | Vercel (前端) + Railway/自建 (后端) | 低成本快速上线 |

## 8. 数据库设计

详见 [design.md](./system-design.md)。

核心设计原则：

- **分层存储**：`notice_content`（原始）→ `cleaned_content`（清洗）→ `notice_summary`（摘要）→ `notice_tag`（标签），逐级降本
- **AI 状态机**：7 态设计（0→1→2→3→4, -1, -2），支持断点续跑
- **标签行级存储**：`notice_tag` 表走 B-Tree 索引，避免 JSON 路径查询
- **资质独立表**：公司资质和人员资质独立管理，支持到期预警
- **匹配结果表**：逐项匹配详情 + 扣分 + 推荐等级

## 9. MVP 范围与里程碑

### Phase 1: 数据基座 + AI Pipeline（Week 1）

**目标：知了 API 数据入库 → AI 提取 → 匹配 → 企微推送，跑通全链路**

- [ ] Supabase 建表迁移（bidding_notice + notice_tag + 资质表 + 匹配结果表）
- [x] 知了标讯 API 对接：Node.js 定时任务，每天 12:00 和 23:00 拉取广东省 IT 运维类公告
- [ ] mimo-v2.5-pro AI Pipeline：清洗 → 摘要 → 结构化打标
- [ ] 公司资质 + 人员资质数据录入
- [ ] 匹配引擎（规则引擎）：资格比对 + 扣分计算 + 推荐等级
- [ ] 企微群机器人 webhook 推送

**交付物：** 企微群每天收到匹配的标讯推送

### Phase 2: PWA 前端（Week 2-3）

**目标：销售能用手机查看标讯和匹配结果**

- [ ] React + Vite + shadcn/ui + Tailwind CSS 项目搭建（含移动端响应式适配）
- [ ] 标讯列表页（按推荐等级排序、筛选）
- [ ] 标讯详情页（结构化信息 + 匹配结果 + 扣分明细）
- [ ] 资质管理页（公司 + 人员 CRUD）
- [ ] PWA 配置（manifest.json + Service Worker + 安装引导）
- [ ] 关键词搜索

**交付物：** 可安装到手机主屏幕的 PWA 应用

### Phase 3: 打磨 + 扩展（Week 4+）

- [ ] 销售标注功能（已跟进/忽略/已投标）
- [ ] 中标公告录入和回顾分析
- [ ] CLI 工具（供 AI Agent 使用）
- [ ] 标签筛选优化
- [ ] 企业自建平台爬虫（选 1-2 个低反爬平台先试点）
- [ ] 资质到期预警
