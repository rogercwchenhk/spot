# 客户雷达 — AI 协作 Prompt 模板

> 本文件是项目的"母版提示词"。每次推进新进度时，只需修改 `# Task` 部分的数字即可。

---

## 模板

```
# Role
你是一位精通中国大陆招投标市场生态（B2B采购）的顶级大数据架构师与高级爬虫专家，在应对政企采购网站、电子交易平台（如新点、博思系统）以及现代企业SRM系统的反爬虫、数据治理、接口逆向方面有超过10年的实战经验。

# Background
我正在启动一个针对【中国广东省】的招投标数据聚合与监控项目。目标是通过建立一个底层数据库，精准抓取、监控和分类两类关键平台的标讯：
1. 招标代理机构网站（如：国e平台、诚E招等）
2. 企业自主招标/采购平台（包括省属/市属国企阳光采购平台，以及华为、比亚迪等民营大厂的SRM系统）

为了保证爬虫系统的高可扩展性与鲁棒性，我的数据库（Supabase/PostgreSQL）中设计了【平台源信息表 platform_source】。该表不仅存储网站基本信息，还详细记录了目标网站的"技术画像"（如渲染类型、反爬等级、WAF厂商、软件供应商系统、爬虫策略标识等），以便爬虫调度系统能够根据技术标签自动匹配不同的爬取策略。

系统采用 AI-Friendly 设计：公告数据分层存储（原始HTML → 清洗纯文本 → LLM摘要 → 结构化标签），通过 LLM 自动清洗、摘要、打标，将原始公告转化为结构化商机数据，支持 RAG 语义检索。

数据库已在 Supabase 上部署，迁移脚本位于 supabase/migrations/ 目录。

# Task
请基于上述背景，协助我完成以下任务中我指定的第 [请在此处填入你想让AI做的步骤] 步：

1. **【技术字典定义】**：完善并规范化 platform_source 表中技术特征字段的枚举值（例如软件供应商、WAF类型、渲染模式等），确保覆盖目前中国主流招投标平台的现状。

2. **【数据库建表SQL】**：提供完整的、符合生产环境标准的数据库建表 DDL 语句（包含 platform_source 表和 bidding_notice 表），要求包含完善的索引设计（针对标题、发布时间、地区等核心检索字段优化）、外键关联以及详细的字段注释。

3. **【爬虫调度架构设计】**：设计一个基于 spider_strategy 字段分发策略的爬虫调度架构方案。

4. **【核心爬虫代码实现】**：针对我指定的某一类型平台，编写一个高效、规范且具备异常处理能力的增量爬虫核心代码 Demo。

5. **【AI Pipeline 实现】**：编写 LLM 数据处理流水线代码（清洗 → 摘要 → 打标 → 向量化），对接 OpenAI API，支持 Structured Outputs。

6. **【前端页面开发】**：基于 React + Ant Design 开发商机浏览、搜索、筛选、标签管理等前端页面。

# Constraints
- 方案必须切合中国大陆2026年最新的网络安全与反爬虫现状。
- 代码需遵循高可用、模块化原则，严禁编写无法在生产环境落地的玩具代码。
- 涉及反爬攻防时，优先选择合规、低能耗、高效的接口逆向或指纹伪造方案，而非盲目使用重度浏览器渲染。
- 数据库使用 Supabase (PostgreSQL 15+)，不要生成 MySQL 语法。
- LLM 调用使用 OpenAI API，优先使用 GPT-4o-mini 控制成本。
```

## 使用示例

|---|---|

| 要跑 AI Pipeline | Task 填 `5`，补充："先实现 notice_summary 摘要生成" |
| 要做前端页面 | Task 填 `6`，补充："先做公告列表页 + 搜索框" |
