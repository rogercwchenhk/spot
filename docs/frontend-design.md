# 客户雷达 — 前端设计文档

> 版本: v2.0 | 日期: 2026-07-06 | 基于实施计划 v2.2
> 状态: Phase 5 完成（前端全面测试 + 安全加固 + 移动端适配），Backlog B1-B4 已交付

---

## 1. 设计原则

- **信息密度优先**：桌面端充分利用屏幕空间，表格/列表密集展示
- **移动端友好**：关键操作在手机上可达，PWA 可安装到主屏幕
- **国际化视觉**：shadcn/ui 质感，克制配色，大量留白，清晰层级
- **响应式断点**：`sm: 640px` / `md: 768px` / `lg: 1024px` / `xl: 1280px`

## 2. 认证与权限

### 2.1 角色定义

| 角色 | 能力 | 适用人群 |
|---|---|---|
| admin | 所有页面 + 资质增删改 + 平台管理 + 推送配置 | 管理者 |
| viewer | 标讯列表/详情/搜索（只读）+ 资质查看（只读）+ 合同查看（只读）+ 报表查看 + 通知查看 | 销售人员 |

### 2.2 认证流程

\`\`\`
用户访问任意页面
      │
      ├─ 已登录 ──→ 正常访问，按角色展示内容
      │
      └─ 未登录 ──→ 跳转登录页
\`\`\`

- 使用 Supabase Auth 内置的邮箱密码登录
- 登录后 Supabase 返回 JWT，前端存储在 localStorage
- API 请求自动携带 Authorization header
- RLS 策略按角色控制数据读写权限
- API 请求返回 401 时自动清除 session 并跳转登录页

## 5. 组件规范

### 4.1 推荐等级配色

| 等级 | 颜色 | Tailwind Class | 含义 |
|---|---|---|---|
| strong | 绿色 | `text-green-600 bg-green-50` | 强推 |
| yes | 蓝色 | `text-blue-600 bg-blue-50` | 可以投 |
| risky | 橙色 | `text-orange-600 bg-orange-50` | 有风险 |
| no | 红色 | `text-red-600 bg-red-50` | 不建议 |

### 4.2 状态徽章

| 状态 | 颜色 | Tailwind Class |
|---|---|---|
| 有效/已接入 | 绿色 | `bg-green-100 text-green-800` |
| 即将到期/待对接 | 黄色 | `bg-yellow-100 text-yellow-800` |
| 过期/已放弃 | 红色 | `bg-red-100 text-red-800` |

### 5.3 组件实现（自定义 Tailwind 组件，非 shadcn/ui）

| 页面元素 | 实现方式 |
|---|---|
| 筛选 Tab | 按钮组 + 状态切换 (Tailwind) |
| 卡片列表 | div + border + hover 状态 |
| 表格 | HTML table + Tailwind (资质/合同/平台) |
| 弹窗表单 | Portal Modal + 表单校验 (自定义 Modal 组件) |
| 分页 | 按钮组 (上一页/页码/下一页) |
| 搜索框 | input + lucide 图标 |
| 徽章 | span + Tailwind badge 类 |
| 开关 | button toggle (自定义) |
| 通知 | Toast hook + Portal 渲染 |
| 骨架屏 | animate-pulse + bg-slate-100 |
| 通知中心 | NotificationBell 组件 (Bell icon + 下拉面板) |
| 离线提示 | useOnlineStatus hook + 顶部 amber 条 |
| 错误兜底 | ErrorBoundary (class component) |

## 11. 未来扩展路线（暂不实现，架构预留）

### 11.1 多后端对接（800客 CRM）

当前前端只对接 Customer Radar 后端。未来需要接入 800客 CRM 系统时，前端架构需支持多后端：

```
组件 → hooks/useApi() → apiClient
                           ├── radarApi   (本项目 Express:3200)
                           ├── crmApi     (800客 CRM)
                           └── 共享 auth token 管理
```

**架构预留要点（搭建时即遵循）：**
- API 调用统一走 `src/lib/api/` 抽象层，组件不直接 fetch
- 每个后端独立的 `baseURL` + 请求/响应拦截器，通过 `.env` 配置
- 认证 token 按后端分别管理（Supabase JWT vs 800客自有 token）
- React Query / SWR 缓存按后端分域，避免数据串台
- 后续 CRM 合同数据可通过后端同步给匹配引擎（类似 Cato CMS sync 模式）

**实施时机：** 当 800客 CRM API 对接需求确认后，增加 `crmApi` 配置即可，页面组件无需改动。

### 11.2 Capacitor 原生 App（拍照/定位/扫码）

当前 PWA 能覆盖基础场景：
- 拍照/选相册：`<input capture>` + MediaDevices API
- 定位：Geolocation API
- 扫码：`html5-qrcode` 等 JS 库

当需要原生能力时（后台定位、蓝牙、iOS 原生推送、通讯录等），在现有 React + Vite 前端上加 Capacitor 层：

```
现在: React + Vite → 浏览器 PWA
未来: React + Vite + Capacitor → iOS/Android App（同一套代码）
```

**架构预留要点（搭建时即遵循）：**
- 摄像头/定位/扫码功能抽成独立 hooks（`useCamera` / `useGeolocation` / `useScanner`）
- 不硬编码浏览器特有 API，保留替换为 Capacitor Plugin 的空间
- 静态资源引用用相对路径，不用 `/` 绝对路径（Capacitor WebView 兼容）

**实施时机：** 当业务需要原生功能时，`npm install @capacitor/core @capacitor/cli` + 对应 Plugin 即可，不重写前端。
